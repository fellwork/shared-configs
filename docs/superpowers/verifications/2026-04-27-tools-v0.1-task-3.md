## Builder deviations from spec

### Deviation 1: Import order (`readdirSync` before `readFileSync`)

**Judgment: Justified — cosmetic, tooling-enforced.**

Biome's import sorting rule requires specifiers in a named import to appear in alphabetical order. `readdirSync` sorts before `readFileSync` alphabetically, matching Biome's requirement. The spec listed them in a different order, but Biome is an enforced gate in this repo (`bunx biome check .` is part of `tools:ci`). Not reordering would cause the check to fail. No behavioral impact.

### Deviation 2: TypeScript strict narrowing (`isPrivate: false` instead of `isPrivate: manifest.private === true`)

**Judgment: Justified — semantically equivalent at that code point, TS-enforced.**

After `if (manifest.private === true) continue`, TypeScript's control-flow narrowing correctly infers that `manifest.private` can only be `false | undefined` at the `result.push(...)` site. The expression `manifest.private === true` becomes a tautological `false`, which TS (under strict mode with `exactOptionalPropertyTypes`) rejects as a type error. Using the literal `false` is semantically identical: we only ever push packages where `private !== true`, so `isPrivate` will always be `false` for any package that reaches this point. The spec's intent (expose `isPrivate` on the returned object) is honored.

### Deviation 3: Root `tsconfig.json` modified (added `moduleResolution: "Bundler"`, `module: "ESNext"`, `allowImportingTsExtensions: true`)

**Judgment: Justified — necessary for spec compliance, contained to non-emitting scope.**

The spec requires the test file to import `'./discover-packages.ts'` with an explicit `.ts` extension (Bun-native style). TypeScript rejects `.ts` extension imports without `allowImportingTsExtensions: true`. The root `tsconfig.json` covers `tools/**/*.ts` and has `noEmit: true`, which is a prerequisite for `allowImportingTsExtensions` — TypeScript explicitly requires either `noEmit` or `isolatedModules` when this flag is set. The `moduleResolution: "Bundler"` and `module: "ESNext"` additions are required companion settings for `allowImportingTsExtensions`. The root tsconfig is NOT used by any package that emits — each package (`release-config`, `tsconfig`, etc.) has its own `tsconfig.json` that extends their own base config. The cascade risk was fully verified: `bunx tsc -b` in `packages/release-config` is clean.

## Concerns / observations

1. **Sort is by directory path, not package name.** The `expandWorkspaceGlob` function sorts directory paths (line 46). The spec says "deterministic order (alphabetical via sort)". The test only verifies idempotence (`r1 === r2`), not that the sort is by name. In practice, for `@fellwork/*` packages, the directory names (e.g., `biome-config`, `tsconfig`) match the package name suffixes, so the output is alphabetical by name. A future consumer relying on strict name-alphabetical ordering could be surprised if directory names diverge from package name suffixes. This is worth noting for the spec, but does not constitute a failure — determinism is guaranteed, and the real-world output is correctly ordered.

2. **Test 9 (lines 121–134) naming vs. behavior.** The test is named `'sets isPrivate=true for private packages when discovered separately'` but the body only verifies `result[0]?.isPrivate` is `false` for a non-private package. The test name implies private packages can have `isPrivate: true`, which is not true (private packages are filtered out entirely). The test name is slightly misleading. Not a correctness issue — the assertion is sound — but the test name could mislead a future reader.

3. **`allowImportingTsExtensions` in root tsconfig is a permanent addition.** This flag is now required for `bunx tsc --noEmit` to stay clean whenever any `tools/lib/*.test.ts` imports `./module.ts` with explicit extension. Future tooling additions under `tools/lib/` must follow the same Bun-native import style or the flag will be superfluous. Consistent — not a problem — but worth awareness.
