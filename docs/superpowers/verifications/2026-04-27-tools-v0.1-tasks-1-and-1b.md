## Builder deviations from spec

### Deviation 1: `validate-kinds` command changed from `'bun run validate-kinds.ts'` to `'bun validate-kinds.ts'`

**Judgment: Justified — with a caveat about local reproduction.**

Builder's stated rationale: bun 1.1.42 (the moon toolchain pin) interprets `bun run <name>.ts` as a package.json script lookup, not direct file execution. Since `tools/` has no `package.json`, the script is not found.

Verifier findings:
- `.moon/toolchain.yml` confirms bun is pinned at `1.1.42` (verified).
- Local system bun is `1.3.13`. At this version, `bun run validate-kinds.ts` run from `tools/` **succeeds** (exit 0, all 6 manifests valid). This means the regression cannot be reproduced locally in this environment.
- Moon executes tasks using the toolchain-pinned bun (1.1.42) rather than the system bun, so the behavior in CI/moon context is expected to differ from local execution. Builder's claim about 1.1.42 behavior is credible and consistent with documented bun version history: `bun run <file>.ts` was not reliably treated as direct file execution in 1.1.x without a matching package.json script.
- The fixed form `bun validate-kinds.ts` runs correctly at both local (1.3.13) and the moon-invoked form (confirmed via `bunx moon run tools:validate-kinds` — PASS).
- The deviation is therefore a correct defensive fix: `bun validate-kinds.ts` is unambiguous direct file execution in all bun versions, whereas `bun run validate-kinds.ts` has version-dependent script-lookup semantics.
- **Note for Team Lead awareness:** Verifier cannot reproduce the original failure locally because the system bun is newer than the toolchain pin. The claim is credible but would ideally be confirmed by a CI run under the pinned 1.1.42 binary, or by installing bun 1.1.42 explicitly.
