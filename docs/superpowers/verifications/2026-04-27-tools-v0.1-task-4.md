## CRITICAL: uses: traversal logic

### Overview

The traversal is implemented as two cooperating functions:

- `extractLocalWorkflowRefs(content)` (lines 57–68): extracts local `uses:` refs using regex `/uses:\s+\.\/\.github\/workflows\/([^\s#]+)/`. Skips comment lines (`#`). Returns only the filename (not the full path).
- `collectWorkflowFiles(workflowsDir, entryName, visited)` (lines 73–92): recursive collector, returns list of filenames to inspect.

### Local relative refs (`./.github/workflows/x.yml`)

**PASS.** The regex `uses:\s+\.\/\.github\/workflows\/([^\s#]+)` precisely targets the `./` prefix pattern. Confirmed working against `release.yml` → `npm-release.yml` (real repo). Adversarial test also confirmed: `release.yml` using `inner.yml` correctly pulls in `inner.yml`'s content.

### Remote/absolute refs (`org/repo/.github/workflows/x.yml@v1`, `actions/checkout@v4`, etc.)

**PASS.** The regex requires the literal `./` prefix — it does not match refs without `./`. In `npm-release.yml`, the following external refs are present:
- `uses: actions/checkout@v4`
- `uses: fellwork/shared-configs/.github/actions/setup-bun-biome@v1`
- `uses: actions/setup-node@v4`
- `uses: changesets/action@v1`

None of these match the regex. All are safely ignored. No network fetch is attempted.

### Missing target file

**PASS.** `collectWorkflowFiles` (line 83) returns `[entryName]` (the name of the missing file) when `existsSync` returns false. `checkWorkflowFile` is then called on that path, also detects the missing file via `existsSync`, and returns `{ ok: false, reasons: ['workflow file not found: <name>'] }`. Verified empirically: a `release.yml` referencing a non-existent `missing-workflow.yml` produces:

```
@org/foo  FAIL
  - workflow file not found: missing-workflow.yml
  - missing 'id-token: write' permission (required for OIDC) in workflow chain
Exit code: 1
```

No crash — clean FAIL.

### Recursion guard

**PASS.** `collectWorkflowFiles` takes a `visited = new Set<string>()` (line 77) as an accumulating parameter. Line 78 short-circuits immediately if `entryName` is already in the set. Line 79 adds it before recursing. Verified empirically with a deliberate cycle (`release.yml` → `b.yml` → `release.yml`): tool completed without timeout, reported results correctly, no infinite loop.

### Summary

All four edge cases are correctly handled. The traversal logic is sound.

## Builder deviations from spec

### Deviation 1: `uses:` reference traversal added

**Judgment: JUSTIFIED.**

The spec's `checkWorkflow` template was written for a workflow with direct `permissions:` and steps. The actual `release.yml` is a thin dispatcher containing only `uses: ./.github/workflows/npm-release.yml` — it has no `permissions:` block. Without traversal, the auditor would:
1. Find no `id-token: write` in `release.yml` → FAIL
2. Potentially find no NPM_TOKEN references → partial pass

All 9 packages would be incorrectly flagged as FAIL. The traversal is not "extra feature creep" — it is required for the tool to be correct against the actual repository layout.

The implementation is conservative: it only follows local `./` refs (cannot fetch remote), has a recursion guard, and handles missing files gracefully. No spec constraints are violated.

### Deviation 2: `--repo-root` default changed from `process.cwd()` to `resolve(__dirname, '..')`

**Judgment: JUSTIFIED.**

When moon runs `tools/trust-audit.ts` as a task in the `tools` project, its working directory is the `tools/` directory, not the repo root. `process.cwd()` would resolve to `.../shared-configs/tools/`, causing `discoverPackages` to fail (no `package.json` with `workspaces` in `tools/`). The fix matches the established pattern from `audit-tools-shape.ts` (as noted in the manifest). This is a necessary correction to the spec template, not a scope expansion.

Low-severity since both files are in the same repo and commits will typically touch the task or other inputs. Recommend adding `/.github/workflows/npm-release.yml` to inputs in a follow-up.
