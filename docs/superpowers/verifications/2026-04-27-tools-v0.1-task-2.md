## Existing-tool modifications (Iron Law check)

### `tools/validate-kinds.ts`

Full diff from baseline `1577888`:

```
+++ Exit codes doc comment appended to existing JSDoc (lines 6–8 in new file):
+  * Exit codes:
+  *   0 - all manifests valid
+  *   1 - one or more manifests failed validation

+++ import { parseArgs } from 'node:util' added

+++ Inside main(), before any logic:
+  // No CLI flags; parseArgs is here per tools/ convention §2.g
+  parseArgs({ args: process.argv.slice(2), options: {} })
```

**Judgment: COSMETIC + CONVENTION-CONFORMANCE with one behavioral side-effect.**

The doc comment addition is purely cosmetic. The `parseArgs` import and no-op call is the convention-conformance addition the Builder claimed. However, `parseArgs({ options: {} })` is not a true no-op with respect to input behavior: it will throw `ERR_PARSE_ARGS_UNKNOWN_OPTION` (exit 1 via uncaught exception) if any unknown CLI flag is passed. Before this change, unknown flags were silently ignored (no `parseArgs` existed at all).

**In practice, this does not regress any current caller:** `validate-kinds` is invoked only as `bun validate-kinds.ts` (no flags) in `tools/moon.yml`, and there are no GitHub workflow files that invoke it with any flags. The normal invocation path is unaffected.

**Verdict:** Iron Law boundary is at the edge. The spec says "adding a no-op parseArgs call is acceptable (cosmetic + convention-conformance)." The call is structurally a no-op for all current callers. The behavioral change (rejection of unknown flags) is a latent side-effect but does not affect any existing invocation. Recorded here as a note for the Team Lead; not classified as a violation.

## Concerns / observations

1. **`validate-kinds.ts` `parseArgs` strict-mode side-effect.** The added `parseArgs({ options: {} })` call means any invocation with unknown CLI flags will now throw (exit 1 via unhandled exception) rather than silently ignoring them. All current callers pass no flags, so this does not break anything today. However, it is not a true behavioral no-op. Team Lead should be aware if future callers or CI scripts attempt to pass flags (e.g., `--help`).

2. **lib/-skip via non-recursive readdir.** The auditor's lib/-skip works by listing only direct children of `tools/` (non-recursive). This correctly excludes `tools/lib/` because it is a directory. The spec says to skip `tools/lib/*.ts` files — the implementation satisfies this via the directory-not-a-file filter, not an explicit path prefix filter. Future-proofing note: if a file named `lib-utils.ts` were added directly to `tools/`, it would NOT be skipped (spec doesn't require this). This is in-spec behavior.

3. **moon cache state.** During verification, `bunx moon run tools:ci` returned cached results from prior runs (5 cached, 280ms). This is expected behavior and does not indicate a freshness concern — cache was populated with the current HEAD results.

4. **Manifest observation.** The manifest (commit `71d32c6`) accurately documents all files modified including the existing-tool changes. The observation about "files were already present in the working tree when this task was executed" is a housekeeping note and does not affect correctness.
